## 💡 核心设计哲学

### 最高原则：任务拆解原则

**核心理念**：
> **"不要把太复杂的任务交给 LLM"**
> 
> **"把大任务拆分成多个小的、易于实现的、易于理解的任务"**
> 
> **"稳定输出 > 智能发挥"**

**为什么**：
- ❌ 复杂任务 → LLM 难以理解 → 结果不可控
- ✅ 简单任务 → LLM 轻松完成 → 结果稳定

**如何拆解**：
1. 识别复杂任务（长文本、精准定位、多步推理）
2. 使用工具辅助（知识库、Python 正则、智能表格）
3. LLM 只做擅长的事（理解语义、整理格式、简单推理）

---

### 原则 1: 检索增强模式

**核心理念**：
> "不要高估 LLM 的能力，善用工具辅助"
> 
> "先缩小范围 → 再让 LLM 处理，而不是把一切丢给 LLM"

**适用场景**：
- 长文档信息提取
- 复杂文档分析
- 知识库问答
- 多文档处理

**标准流程**：
```
输入文档 → 添加知识（建立索引） → 知识检索（缩小范围） → LLM 处理（精准提取） → 输出
```

**优势**：
- ✅ 准确性更高
- ✅ 算力成本更低
- ✅ 可解释性更强

---

### 原则 2: 模板化 + 查表法

**核心理念**：
> "预定义模板 > 动态生成"
> 
> "查表法 > LLM 生成"
> 
> "可积累 > 一次性"

**适用场景**：
- 提示词生成
- 文档模板处理
- 格式化输出
- 配置管理
- 规则引擎

**标准流程**：
```
1. 创建智能表格（存储模板/规则/提示词）
2. 输入处理：识别关键字/参数
3. 智能表格检索：查找预定义模板
4. Python 拼接：组合成完整结果
5. 输出
```

**优势**：
- ✅ 100% 稳定性
- ✅ 易于维护
- ✅ 可积累
- ✅ 高质量

---

### 原则 3: LLM 依赖度评估

**核心理念**：
> "能用工具解决的，不用 LLM"

**LLM 依赖度评估表**：

| 任务类型 | 推荐方案 | LLM 依赖度 | 建议 |
|---------|---------|----------|------|
| 正则匹配、结构化提取 | Python 代码 | 低 | ✅ 优先 |
| 关键词匹配 | 关键词识别节点 | 低 | ✅ 优先 |
| 预定义模板生成 | 智能表格查表法 | 低 | ✅ 优先 |
| 短文档简单提取 | 直接 LLM | 中 | ⚠️ 可接受 |
| 长文档复杂提取 | 检索增强 + LLM | 中 | ⚠️ 可接受 |
| 复杂推理、创意生成 | LLM | 高 | ❌ 慎用 |

**评估原则**：
- 能用 Python/正则解决的，不用 LLM
- 能用关键词识别解决的，不用 LLM
- 能用预定义模板解决的，不用 LLM 动态生成
- 必须用 LLM 时，优先检索增强模式

## 🎓 设计哲学总结

### 核心思想

**"稳定输出 > 智能发挥"**

这不是否定 LLM 的价值，而是：
- 让 LLM 做它擅长的
- 用工具弥补 LLM 的不足
- 构建稳定、可控、可维护的系统

### 工程化思维

1. **可预测性**：系统行为应该是可预测的，不是黑盒
2. **可维护性**：系统应该易于修改、优化、调试
3. **可积累性**：系统应该能够持续积累知识和经验
4. **稳定性**：系统输出应该稳定可靠，不是每次都不一样

### LLM 的正确使用方式

**LLM 是工具，不是万能钥匙**

- ✅ 让 LLM 处理它擅长的（理解语义、整理格式、简单推理）
- ❌ 不让 LLM 做它不擅长的（精准定位、长文本理解、稳定输出）

**LLM 是助手，不是决策者**

- ✅ LLM 辅助人工，提高效率
- ❌ 不让 LLM 完全自主决策

## 📚 学习历程

### 案例 1：文档分数提取

**需求**：从用户上传的投标文档中提取商务分、技术分、价格分

**初始方案**（错误）：
```
输入文档 → 文档解析 → LLM（整篇文档） → 输出
```
问题：LLM 容易遗漏，算力浪费，准确性低

**优化方案**（检索增强）：
```
输入文档 → 添加知识 → 知识检索 → LLM（检索片段） → 输出
```
优势：LLM 只处理相关片段，准确性大幅提升

**学到的**：不要高估 LLM 的能力，善用工具辅助

---

### 案例 2：动态提示词生成

**需求**：根据模板文档的关键字，生成用于提取这些关键字的提示词

**初始方案**（错误）：
```
输入模板 → LLM 识别关键字并生成提示词 → 输出
```
问题：提示词不稳定、质量不可控、难以维护

**优化方案**（模板化 + 查表法）：
```
1. 创建智能表格（关键字 - 提示词映射表）
2. Python 正则识别关键字
3. 智能表格检索 → 查找预定义提示词
4. Python 拼接 → 组合成完整提示词
```
优势：100% 稳定、易于维护、可积累

**学到的**：预定义模板 > 动态生成，查表法 > LLM 生成

---

### 案例 3：合同信息提取

**需求**：从合同中提取所有关键信息

**初始方案**（错误）：
```
整份合同 → LLM"提取所有信息" → 输出
```
问题：合同条款多，LLM 容易遗漏

**优化方案**（任务拆解）：
```
任务 1：合同分段（添加知识）
任务 2：检索各方信息（知识问答）→ LLM 提取
任务 3：检索金额信息（知识问答）→ LLM 提取
任务 4：检索日期信息（知识问答）→ LLM 提取
任务 5：合并结果（Python 拼接）
```
优势：每个任务都简单、可控、稳定

**学到的**：把大任务拆分成多个小任务，稳定输出 > 智能发挥

## 🚀 应用指南

### 何时使用这些原则

**判断标准**：
1. 任务是否涉及长文本（>1000 字）？
2. 是否需要精准定位信息？
3. 是否需要稳定输出（不是创意生成）？
4. 是否需要易于维护和优化？

**如果以上任一为"是"，请使用这些原则**

### 实施步骤

1. **识别复杂任务**
   - 长文本理解
   - 精准定位
   - 复杂推理
   - 动态生成

2. **选择拆解方法**
   - 长文本 → 检索增强
   - 精准匹配 → Python 正则
   - 模板生成 → 智能表格查表
   - 复杂推理 → 分步处理

3. **验证每个小任务**
   - 确保每个节点都可测试
   - 确保每个节点都可验证
   - 确保每个节点都易于调试

## 📝 总结

> 复杂任务别硬上，
> 拆解分成小任务。
> 长文本，用检索，
> 精准匹配用正则。
