# 工作流搭建指南

## 📋 概述

本指南帮助你使用平台提供的工作流节点搭建自动化工作流。工作流由多个节点组成，通过节点间的连接实现复杂的业务逻辑。

---

## 🎯 工作流基础

### 工作流结构

一个完整的工作流包含:

1. **输入节点**: 定义工作流的输入变量
2. **处理节点**: 执行具体的业务逻辑
3. **输出节点**: 输出工作流的最终结果

### 连接方式

- **线性连接**: 节点 A → 节点 B → 节点 C
- **条件分支**: 根据条件执行不同的分支路径
- **并行处理**: 同时执行多个并行节点
- **循环处理**: 对数组中的每个元素执行相同操作
- **路径合并**: 多个分支自动合并到下一个节点

### 数据传递机制

**变量引用规则**:

1. **输入变量**: 由用户在工作流启动时指定
   - 例如：`input_text`, `user_query`, `data_list`

2. **节点输出变量**: 每个节点执行后会创建一个变量
   - 变量名：由节点配置时指定
   - 其他节点可以直接选择使用这些变量

3. **变量使用**: 后续节点配置时可以直接选择已创建的变量
   - 例如：LLM 节点的 prompt 可以选择 `${input_text}` 作为输入

## 🔧 通用配置说明

### 输入输出变量命名规范

1. **变量类型前缀**:
   - `文本：variable_name` - 文本类型
   - `json: variable_name` - JSON 对象
   - `json/数组：variable_name` - JSON 数组
   - `markdown: variable_name` - Markdown 格式
   - `文档：variable_name` - 文档文件

2. **变量命名建议**:
   - 使用英文或拼音命名
   - 语义清晰明确
   - 避免使用特殊字符

### 节点配置流程

1. **选择节点类型** → 根据功能需求选择
2. **配置输入变量** → 指定变量类型和名称
3. **配置节点参数** → 填写具体配置项
4. **配置输出变量** → 指定输出变量类型和名称
5. **连接节点** → 与其他节点连接形成工作流

## 💡 最佳实践

### 设计原则

1. **模块化**: 复杂工作流拆分为子工作流
2. **复用性**: 通用功能封装为独立节点
3. **可读性**: 节点命名清晰，注释完整
4. **容错性**: 添加错误处理和异常分支

### 性能优化

1. **并行处理**: 无依赖关系的节点并行执行
2. **数据过滤**: 尽早过滤不必要的数据
3. **缓存策略**: 重复使用的数据使用缓存
4. **批量操作**: 批量数据使用批量处理节点

### 调试技巧

1. **分步测试**: 逐个节点测试验证
2. **日志记录**: 关键节点添加日志输出
3. **数据检查**: 使用 Python 节点打印中间数据
4. **版本管理**: 保存多个版本便于回滚

## 🚀 高级设计模式

### 模式 0: 任务拆解原则（最高原则）

#### 核心理念

**"不要把太复杂的任务交给 LLM"**

**"把大任务拆分成多个小的、易于实现的、易于理解的任务"**

**"稳定输出 > 智能发挥"**

#### 为什么

| 任务类型 | LLM 表现 | 结果 |
|---------|---------|------|
| 简单任务（处理 200 字） | 轻松完成 | ✅ 稳定 |
| 中等任务（处理 1000 字） | 可能遗漏 | ⚠️ 中等 |
| 复杂任务（处理 10000 字） | 难以理解 | ❌ 不可控 |

**原因**：
- ❌ 复杂任务 → LLM 难以理解 → 结果不可控
- ✅ 简单任务 → LLM 轻松完成 → 结果稳定

#### 如何拆解

**步骤 1：识别复杂任务**

复杂任务的特征：
- 需要多步推理
- 需要理解长文本（>1000 字）
- 需要精准定位信息
- 需要区分多个维度

**步骤 2：找到拆解的方法**

| 复杂任务类型 | 拆解方法 | 使用工具 |
|------------|---------|---------|
| 长文本理解 | 分段 → 检索 → 处理 | 知识库 |
| 精准定位 | 正则匹配 → 整理 | Python |
| 模板生成 | 查表 → 拼接 | 智能表格 |
| 复杂推理 | 分步推理 → 合并 | 多 LLM 节点 |

**步骤 3：验证每个小任务**

确保每个节点都：
- ✅ 任务单一明确
- ✅ 输入输出清晰
- ✅ 可测试可验证
- ✅ 易于调试优化

#### 拆解示例

**示例 1：文档分数提取**

**复杂任务**："从这篇 10000 字的投标文档中提取商务分、技术分、价格分"

**错误做法**：
```
整篇文档 → LLM"提取三个分数" → 输出
问题：
  - LLM 需要理解 10000 字
  - 需要精准定位分数位置
  - 需要区分三个维度
  - 结果：容易遗漏、不稳定
```

**正确做法（拆解任务）**：
```
任务 1：文档分段和索引（添加知识节点）
  - 简单任务：自动分段、向量化
  - 工具：知识库
  
任务 2：检索相关片段（知识问答节点）
  - 简单任务：根据问题检索最相关的片段
  - LLM 只需处理 200-500 字，不是 10000 字
  
任务 3：提取分数（LLM 节点）
  - 简单任务：从检索到的片段中提取分数
  - 范围明确、内容聚焦
  
任务 4：格式化输出（数据类型转换节点）
  - 简单任务：文本转 JSON

每个任务都简单、可控、稳定！
```

**示例 2：动态提示词生成**

**复杂任务**："根据模板文档的关键字，生成用于提取这些关键字的提示词"

**错误做法**：
```
输入模板 → LLM"识别关键字并生成提示词" → 输出
问题：
  - LLM 可能遗漏关键字
  - 生成的提示词质量不稳定
  - 无法积累和优化
```

**正确做法（拆解任务）**：
```
任务 1：识别关键字（Python 正则节点）
  - 简单任务：用正则匹配关键字格式
  - 100% 准确，不会遗漏
  
任务 2：查找提示词模板（智能表格检索节点）
  - 简单任务：根据关键字查表
  - 获取预定义的优质提示词
  
任务 3：拼接提示词（Python 拼接节点）
  - 简单任务：字符串拼接
  - 组合成完整提示词

每个任务都简单、可控、稳定！
```

**示例 3：合同信息提取**

**复杂任务**："从这份合同中提取所有关键信息"

**错误做法**：
```
整份合同 → LLM"提取所有信息" → 输出
问题：
  - 合同条款多，LLM 容易遗漏
  - 信息类型复杂，难以区分
```

**正确做法（拆解任务）**：
```
任务 1：合同分段（添加知识节点）
  - 简单任务：按条款分段
  
任务 2：检索各方信息（知识问答节点）
  - 简单任务：检索"甲方"、"乙方"相关片段
  - LLM 提取各方信息
  
任务 3：检索金额信息（知识问答节点）
  - 简单任务：检索"金额"、"价款"相关片段
  - LLM 提取金额信息
  
任务 4：检索日期信息（知识问答节点）
  - 简单任务：检索"日期"、"期限"相关片段
  - LLM 提取日期信息
  
任务 5：合并结果（Python 拼接节点）
  - 简单任务：JSON 合并

每个任务都简单、可控、稳定！
```

#### 设计要点

1. **识别不可控的任务**
   - 长文本理解（>1000 字）
   - 精准定位（大海捞针）
   - 复杂推理（多步推理）
   - 动态生成（提示词、模板）

2. **找到拆解的方法**
   - 长文本 → 知识库分段 → 检索 → LLM
   - 精准匹配 → Python 正则 → LLM 整理
   - 模板生成 → 智能表格 → 查表 → 拼接
   - 复杂推理 → 分步处理 → 合并结果

3. **验证每个小任务**
   - 确保每个节点都可测试
   - 确保每个节点都可验证
   - 确保每个节点都易于调试

#### LLM 擅长 vs 不擅长

**LLM 擅长的（让 LLM 做）**：
- ✅ 理解语义（短文本）
- ✅ 整理格式
- ✅ 简单推理
- ✅ 内容生成（创意类）

**LLM 不擅长的（用工具辅助）**：
- ❌ 从长文本中精准定位（用知识库检索）
- ❌ 精准匹配格式（用 Python 正则）
- ❌ 稳定输出模板（用智能表格查表）
- ❌ 复杂多步推理（拆解成多步）

### 模式 1: 检索增强模式（强烈推荐）

#### 核心理念
**"不要高估 LLM 的能力，善用工具辅助"**

当处理长文档或复杂任务时，不要直接把所有内容丢给 LLM，而是：
```
先缩小范围 → 再让 LLM 处理
而不是
一开始就把所有问题丢给 LLM
```

#### 适用场景
- ✅ 长文档信息提取（如从投标文档中提取分数）
- ✅ 复杂文档分析（如法律合同审查）
- ✅ 知识库问答（如技术文档查询）
- ✅ 多文档处理（如简历筛选）

#### 标准流程
```
输入文档 → 添加知识（建立索引） → 知识检索（缩小范围） → LLM 处理（精准提取） → 输出
```

#### 优势对比

| 维度 | 直接丢给 LLM | 检索增强模式 |
|------|------------|-------------|
| LLM 输入 | 整篇文档（10000 字） | 检索片段（200 字） |
| LLM 任务 | "从整篇文章中找分数" | "从这几段中找分数" |
| 准确性 | 中等（可能遗漏） | 高（聚焦精准） |
| 算力消耗 | 高（处理长文本） | 中（处理短文本） |
| 可解释性 | 低（不知道 LLM 看了什么） | 高（知道检索了哪些片段） |

#### 实战案例：文档分数提取

**需求**：从用户上传的投标文档中提取商务分、技术分、价格分

**错误做法**：
```
输入文档 → 文档解析 → LLM（整篇文档） → 输出
问题：LLM 容易遗漏，算力浪费，准确性低
```

**正确做法（检索增强）**：
```
1. 输入节点：文档：document
2. 添加知识节点：
   - 输入：文档：document
   - 输出：知识/数组：knowledge_fragments
   - 功能：自动分段、向量化、建立索引
3. 知识问答节点（知识检索 Agent）：
   - 问题："请找出文档中的商务分、技术分、价格分"
   - 输入：文本：user_query
   - 输出：文本：search_result
   - 功能：基于知识库精准检索相关片段
4. LLM 节点：
   - Prompt："根据以下检索结果，提取三个分数并整理成 JSON 格式：${search_result}"
   - 输出：文本：llm_result
   - 功能：整理格式化检索结果
5. 数据类型转换节点：
   - 输入：文本：llm_result
   - 输出：json：score_result
   - 功能：文本转 JSON
6. 输出节点：json：score_result
```

**输出示例**：
```json
{
  "商务分": 85,
  "技术分": 92,
  "价格分": 78
}
```

#### 类似场景应用

**场景 1：法律文档审查**
```
❌ 错误：整篇合同 → LLM"审查所有条款"
✅ 正确：合同 → 知识库 → 检索"违约条款"、"保密条款" → LLM 逐个审查
```

**场景 2：技术文档问答**
```
❌ 错误：整个 API 文档 → LLM"如何调用接口"
✅ 正确：API 文档 → 知识库 → 检索"认证接口"、"参数说明" → LLM 基于检索结果回答
```

**场景 3：简历筛选**
```
❌ 错误：整份简历 → LLM"评估候选人"
✅ 正确：简历 → 知识库 → 检索"工作经历"、"技能" → LLM 基于检索结果评估
```

**场景 4：病历分析**
```
❌ 错误：整本病历 → LLM"诊断病情"
✅ 正确：病历 → 知识库 → 检索"症状"、"检查结果" → LLM 基于检索结果分析
```

#### 设计要点

1. **知识库是"预处理"工具**：不是替代 LLM，而是增强 LLM
2. **分而治之**：把复杂任务拆解成简单任务，逐个解决
3. **智慧地使用 LLM**：让 LLM 处理它擅长的（理解、推理），而不是让它做苦力（大海捞针）

### 模式 2: 模板化 + 查表法（强烈推荐）

#### 核心理念
**"预定义模板 > 动态生成"**
**"查表法 > LLM 生成"**
**"可积累 > 一次性"**

当需要生成提示词、模板等重复性内容时，不要用 LLM 动态生成，而是：
```
预定义模板 → 智能表格存储 → 查表获取 → 拼接使用
```

#### 适用场景
- ✅ 提示词生成（如根据关键字生成提取提示词）
- ✅ 文档模板处理
- ✅ 格式化输出
- ✅ 配置管理
- ✅ 规则引擎

#### 标准流程
```
1. 准备工作：创建智能表格（存储模板/规则/提示词）
2. 输入处理：识别关键字/参数
3. 智能表格检索：查找对应的预定义模板
4. 拼接处理：组合成完整结果
5. 输出应用
```

#### 优势对比

| 维度 | LLM 动态生成 | 模板化 + 查表法 |
|------|------------|---------------|
| 稳定性 | 低（每次可能不同） | 高（100% 稳定） |
| 可控性 | 低（难以控制质量） | 高（完全可控） |
| 可维护性 | 低（难以调试优化） | 高（易于维护） |
| 准确性 | 中（依赖 LLM 状态） | 高（预定义保证） |
| 成本 | 高（每次都调用 LLM） | 低（一次定义，多次复用） |
| 可积累性 | 低（无法积累） | 高（持续积累） |

#### 实战案例：动态提示词生成

**需求**：根据模板文档中的关键字，生成用于提取这些关键字的提示词

**错误做法**：
```
输入模板 → LLM 识别关键字并生成提示词 → 输出
问题：
  - LLM 每次生成的提示词可能不同
  - 提示词质量不稳定
  - 难以维护和优化
  - 无法积累
```

**正确做法（模板化 + 查表法）**：

**步骤 1：创建智能表格**
```
表格名称：关键字 - 提示词映射表
字段：关键字、提示词模板、适用场景、注意事项、版本

示例数据：
| 关键字 | 提示词模板 | 适用场景 | 注意事项 |
|--------|-----------|---------|---------|
| 【公司名称】 | 请从文档中提取【公司名称】，它通常出现在"甲方："、"乙方："等词语后面，格式为公司全称 | 合同 | 注意区分甲方乙方 |
| {日期} | 请从文档中提取{日期}，它通常出现在文档末尾签署处，格式为 YYYY-MM-DD 或 YYYY 年 MM 月 DD 日 | 通用 | 注意区分签署日期和生效日期 |
| <金额> | 请从文档中提取<金额>，它通常包含数字和"元"、"万元"等单位，注意区分大写金额和小写金额 | 合同、报价单 | 注意币种和单位 |
```

**步骤 2：工作流配置**
```
1. 输入节点：文档：template_doc
2. 文档解析节点：
   - 输入：文档：template_doc
   - 输出：文本：document_text
3. Python 正则识别节点：
   - 输入：文本：document_text
   - 输出：json：keywords_data
   - 功能：用正则识别所有关键字（【】、{}、<> 等）
4. 智能表格检索节点：
   - 输入：json：keywords_data
   - 输出：json/数组：prompt_templates
   - 功能：从智能表格中查找每个关键字的提示词模板
5. Python 拼接节点：
   - 输入：
     - json：keywords_data
     - json/数组：prompt_templates
   - 输出：文本：final_prompt
   - 功能：将所有提示词模板拼接成完整提示词
6. 输出节点：
   - 输出：文本：final_prompt
   - 说明：生成的完整提取提示词
```

```
❌ 错误：LLM 动态生成邮件模板
✅ 正确：创建邮件模板库 → 根据场景查表获取 → 填充变量 → 输出
```

**场景 2：报告格式生成**
```

```

**场景 3：规则引擎**
```
❌ 错误：LLM 动态生成处理规则

```

#### 设计要点

1. **智能表格是配置管理中心**：存储提示词模板、规则、配置
2. **预定义 > 动态生成**：预定义保证质量和稳定性
3. **可积累的知识资产**：每次遇到新关键字就添加到表格，越来越完善
4. **易于维护优化**：智能表格可随时修改，修改后立即可用

### 模式 3: LLM 依赖度评估

在工作流设计时，评估 LLM 依赖程度：

| 任务类型 | 推荐方案 | LLM 依赖度 | 建议 |
|---------|---------|----------|------|
| 正则匹配、结构化提取 | Python 代码 | 低 | ✅ 优先 |
| 关键词匹配 | 关键词识别节点 | 低 | ✅ 优先 |
| 短文档简单提取 | 直接 LLM | 中 | ⚠️ 可接受 |
| 长文档复杂提取 | 检索增强 + LLM | 中 | ⚠️ 可接受 |
| 复杂推理、创意生成 | LLM | 高 | ❌ 慎用 |

**评估原则**：
- 能用 Python/正则解决的，不用 LLM
- 能用关键词识别解决的，不用 LLM
- 必须用 LLM 时，优先检索增强模式
