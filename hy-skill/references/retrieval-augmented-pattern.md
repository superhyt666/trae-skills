# 检索增强模式设计指南

## 📋 概述

本文档详细介绍"检索增强"设计模式，这是从实际案例中总结出的重要工作流设计思想。

---

## 💡 核心思想

### 核心理念

> **"不要高估 LLM 的能力，善用工具辅助"**
> 
> **"先缩小范围 → 再让 LLM 处理，而不是把一切丢给 LLM"**

### 思想来源

这个设计模式来源于一个实际案例：

**需求**：从用户上传的投标文档中提取商务分、技术分、价格分

**错误做法**：
```
输入文档 → 文档解析 → LLM（整篇文档） → 输出
```
问题：
- LLM 需要处理整篇文档（可能 10000+ 字）
- 容易遗漏关键信息
- 算力浪费严重
- 准确性无法保证

**正确做法（检索增强）**：
```
输入文档 → 添加知识（建立索引） → 知识检索（缩小范围） → LLM 处理（精准提取） → 输出
```
优势：
- LLM 只处理检索到的相关片段（200-500 字）
- 聚焦精准，不容易遗漏
- 算力成本大幅降低
- 准确性显著提高

## 🎯 适用场景

### 推荐使用检索增强的场景

| 场景类型 | 具体案例 | 原因 |
|---------|---------|------|
| 长文档信息提取 | 投标文档提取分数 | 文档长、信息分散 |
| 复杂文档分析 | 法律合同审查 | 条款多、需要精准定位 |
| 知识库问答 | 技术文档查询 | 文档多、需要语义检索 |
| 多文档处理 | 简历筛选 | 需要对比多个文档 |
| 专业领域分析 | 病历分析 | 需要定位关键信息 |

### 不需要检索增强的场景

| 场景类型 | 具体案例 | 原因 |
|---------|---------|------|
| 短文档处理 | 短信内容分析 | 文档短，直接处理即可 |
| 简单任务 | 情感分析 | 不需要定位具体片段 |
| 创意生成 | 文案创作 | 需要整体理解 |

## 🏗️ 标准流程

### 完整工作流架构

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  输入   │ -> │ 添加知识 │ -> │ 知识检索 │ -> │  LLM    │ -> │  输出   │
│  文档   │    │ (索引)   │    │ (缩小)   │    │ (提取)  │    │  结果   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
```

### 节点配置详解

#### 1. 输入节点
```
变量：文档：document
说明：用户上传的文档文件
```

#### 2. 添加知识节点
```
输入：文档：document
输出：知识/数组：knowledge_fragments
配置：
  - 目标知识库：预先创建的知识库
  - 深度解析：开启
  - 分段规则：自动分段
功能：
  - 自动将文档切分成合理片段
  - 向量化处理
  - 建立语义索引
```

#### 3. 知识问答节点（知识检索 Agent）
```
输入：文本：user_query
输出：文本：search_result
配置：
  - 问题："请找出文档中的商务分、技术分、价格分"
  - 知识来源：上一步添加的知识库
  - 答复模式：知识检索
功能：
  - 基于语义检索最相关的片段
  - 返回精准的检索结果
```

#### 4. LLM 节点
```
输入：文本：search_result
输出：文本：llm_result
配置：
  - Prompt："根据以下检索结果，提取三个分数并整理成 JSON 格式：${search_result}"
  - 模型：Qwen3-32B
  - 创造性：3 级（偏严谨）
功能：
  - 基于检索结果进行精准提取
  - 整理成要求的格式
```

#### 5. 数据类型转换节点
```
输入：文本：llm_result
输出：json：score_result
配置：
  - 转换目标：JSON
功能：
  - 将文本转换为 JSON 对象
```

#### 6. 输出节点
```
输出：json：score_result
```

## 📊 优势对比

### 直接丢给 LLM vs 检索增强模式

| 维度 | 直接丢给 LLM | 检索增强模式 | 提升 |
|------|------------|-------------|------|
| LLM 输入 | 整篇文档（10000 字） | 检索片段（200 字） | 减少 98% |
| LLM 任务 | "从整篇文章中找分数" | "从这几段中找分数" | 任务简化 |
| 准确性 | 中等（可能遗漏） | 高（聚焦精准） | 显著提升 |
| 算力消耗 | 高（处理长文本） | 中（处理短文本） | 降低 80%+ |
| 可解释性 | 低（不知道 LLM 看了什么） | 高（知道检索了哪些片段） | 完全可追溯 |
| 响应速度 | 慢（处理长文本） | 快（处理短文本） | 显著提升 |

## 🔍 类似场景应用

### 场景 1：法律文档审查

**需求**：审查合同中的关键条款

**错误做法**：
```
整篇合同 → LLM"审查所有条款"
问题：合同长、条款多，LLM 容易遗漏
```

**正确做法**：
```
合同 → 知识库 → 检索"违约条款"、"保密条款"、"付款条款" → LLM 逐个审查
```

### 场景 2：技术文档问答

**需求**：回答"如何调用认证接口"

**错误做法**：
```
整个 API 文档 → LLM"如何调用接口"
问题：文档庞大，LLM 难以定位
```

**正确做法**：
```
API 文档 → 知识库 → 检索"认证接口"、"参数说明"、"调用示例" → LLM 基于检索结果回答
```

### 场景 3：简历筛选

**需求**：评估候选人是否匹配岗位

**错误做法**：
```
整份简历 → LLM"评估候选人"
问题：简历信息分散，评估不准确
```

**正确做法**：
```
简历 → 知识库 → 检索"工作经历"、"项目经验"、"技能栈" → LLM 基于检索结果评估
```

### 场景 4：病历分析

**需求**：根据病历分析病情

**错误做法**：
```
整本病历 → LLM"诊断病情"
问题：病历内容多，关键信息分散
```

**正确做法**：
```
病历 → 知识库 → 检索"症状描述"、"检查结果"、"用药记录" → LLM 基于检索结果分析
```

## 🎓 设计要点

### 1. 知识库的角色

**知识库是"预处理"工具，不是替代 LLM，而是增强 LLM**

- 作用：将长文档切分成片段，建立索引
- 价值：让 LLM 从"大海捞针"变成"池塘捞针"
- 定位：LLM 的助手，不是 LLM 的替代

### 2. 分而治之的思想

**把复杂任务拆解成简单任务，逐个解决**

- 第一步：添加知识（自动分段、索引）
- 第二步：知识检索（精准定位）
- 第三步：LLM 处理（基于检索结果）

### 3. 智慧地使用 LLM

**让 LLM 处理它擅长的（理解、推理），而不是让它做苦力（大海捞针）**

- ✅ LLM 擅长：理解语义、推理分析、整理格式
- ❌ LLM 不擅长：从长文本中精准定位信息

## 📈 实施建议

### 何时使用检索增强

**判断标准**：
1. 文档长度 > 1000 字？
2. 信息是否分散在文档各处？
3. 是否需要精准定位特定内容？
4. LLM 直接处理是否容易遗漏？

**如果以上任一为"是"，推荐使用检索增强**

### 实施步骤

1. **创建知识库**：在平台中预先创建知识库
2. **配置添加知识节点**：设置文档输入和知识库
3. **配置知识检索节点**：设置检索问题
4. **配置 LLM 节点**：基于检索结果进行处理
5. **测试验证**：上传测试文档，验证效果

## 🚀 进阶技巧

### 1. 多轮检索

对于复杂任务，可以进行多轮检索：
```
第一轮：检索"商务分"相关 → LLM 提取商务分
第二轮：检索"技术分"相关 → LLM 提取技术分
第三轮：检索"价格分"相关 → LLM 提取价格分
```

### 2. 并行检索

对于独立的任务，可以并行处理：
```
[检索商务分] → LLM 提取
[检索技术分] → LLM 提取  → 合并结果
[检索价格分] → LLM 提取
```

### 3. 检索 + 验证

对于高准确性要求的场景：
```
检索 → LLM 提取 → Python 验证 → 输出
```

## 📝 总结

### 核心口诀

> 长文档，别硬上，
> 知识库，来帮忙。
> 先索引，再检索，
> LLM，更轻松。

### 价值总结

1. **准确性提升**：LLM 处理聚焦内容，不容易遗漏
2. **成本降低**：减少算力消耗，提升响应速度
3. **可解释性增强**：知道检索了哪些片段，完全可追溯
4. **可维护性提高**：每个节点职责单一，便于调试

### 推广价值

这个设计模式可以应用到所有涉及长文档处理、复杂信息提取的场景，是工作流设计的**最佳实践**！
